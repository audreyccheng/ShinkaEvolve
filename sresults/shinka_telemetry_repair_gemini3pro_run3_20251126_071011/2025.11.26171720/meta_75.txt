# GLOBAL INSIGHTS SCRATCHPAD

The following are global insights about optimization approaches and their effectiveness:

# Analysis of Program Evaluation Results

## Successful Algorithmic Patterns
- **Anchor-Weighted Flow Constraints:** The **Current Best Program (Gen 68)** and **Gen 73** significantly outperformed earlier iterations (Counter Accuracy **0.93** vs ~0.83) by dynamically weighting the influence of flow conservation constraints. Instead of treating all routers as valid conservation targets, Gen 68 scales the flow target's weight using a router "Anchor Score" (average neighbor agreement) and "Balance Score" (`2.0 * anchor * (0.4 + 0.6 * balance)`). This ensures that only reliable parts of the network influence the repair of noisy links.
- **Discrete Hypothesis Competition:** High-performing programs (Gen 64, 68, 71) moved away from continuous value adjustment to selecting from a discrete set of hypotheses (Self, Peer, Flow Target, Zero). **Gen 68** (Score 0.92) constructs a `set(candidates)` and scores each using Gaussian likelihoods, effectively arbitrating "who is lying" rather than averaging errors.
- **Status-Aware Priors:** Explicitly penalizing 'Zero' hypotheses when an interface is inferred to be 'Up' proved critical for precision. **Gen 68** applies a penalty (`score *= 0.15`) to zero-value candidates if the status logic suggests the link is active. This prevented the solver from "repairing" noisy low-traffic links to zero (phantom flows), contributing to the high 0.93 counter accuracy.
- **Cluster-Mass Confidence:** **Gen 68** improved confidence calibration (0.88) by calculating probability mass not just for the winning hypothesis, but for the "cluster" of hypotheses within one standard deviation (`sigma`) of the winner. This robustly captures the certainty of the solution space better than single-point likelihoods.

## Ineffective Approaches
- **Uniform Flow Conservation:** Programs that applied flow conservation logic uniformly across the network (e.g., **Gen 60**, **Gen 61**) plateaued with Counter Repair Accuracy around **0.83**. Without weighting routers by their "trustworthiness" (Anchor Score), these algorithms propagated errors from highly noisy routers onto clean links in an attempt to satisfy global balance.
- **Global Fit for Local Confidence:** Algorithms relying heavily on global router balance or solver goodness-of-fit for confidence scores (e.g., **Gen 63**, **Gen 67**) struggled with calibration (scores ~0.62 - 0.76). A globally balanced router does not guarantee that individual links are correct (as errors can cancel out), leading to overconfident but incorrect predictions.
- **Strict Zero Forcing without Soft Priors:** While **Gen 63** achieved perfect status repair by strictly forcing zero flow on "down" interfaces, its confidence calibration suffered (0.628). The lack of a probabilistic "soft" penalty (like the `* 0.15` prior in Gen 68) meant the model couldn't subtly distinguish between a definitely-down link and a noisy low-traffic link, affecting the granularity of confidence scores.

## Implementation Insights
- **Dynamic Flow Weighting Formula:** A key implementation detail in the **Current Best Program (Gen 68)** is the explicit formula for flow influence: `f_weight = 2.0 * r_info['anchor'] * (0.4 + 0.6 * r_info['balance'])`. This coding pattern ensures that flow conservation targets (`f_val`) only compete with observed data (`s_val`, `p_val`) when the router is surrounded by trusted peers (`anchor`) and is internally consistent (`balance`).
- **Hypothesis Set Construction:** **Gen 68** implements a specific search space construction: `candidates = [s_val, 0.0]`; adds `p_val` if peer exists; adds `f_val` (flow target) if weights permit; and adds `(s_val + p_val) / 2.0` *only* if values are relatively close. This prevents the "average" fallacy on highly divergent links (e.g., one side 0, one side 100) where averaging is physically impossible.
- **Adaptive Sigma Calculation:** The best program uses `sigma = max(ABS_TOL, abs(val) * REL_TOL)` (where `REL_TOL=0.02`). This scales the Gaussian kernel width with traffic magnitude, preventing the model from being too lenient on small flows or too strict on massive flows (100Gbps+), which inherently have larger variance.
- **Winner-Cluster Confidence:** The code in **Gen 68** calculates confidence as `winner_mass / total_mass`, where `winner_mass` accumulates scores of all candidates `h` such that `abs(h - winner) <= sigma`. This acknowledges that if "Self" and "Flow" are very close, they support each other, boosting confidence naturally.

## Performance Analysis
- **The 0.92 Performance Ceiling:** The introduction of "Anchor-Weighted Flow" pushed the Combined Score from a plateau of ~0.83 (Gen 60-63) to **0.92** (Gen 68, 72). This suggests that distinguishing *where* to apply conservation constraints is as important as the constraints themselves.
- **Calibration vs. Accuracy Alignment:** Unlike earlier generations where high accuracy sometimes came with poor calibration (Gen 49), **Gen 68** aligns both metrics (Accuracy 0.93, Calibration 0.88). This indicates that the "Cluster Mass" confidence metric is correctly modeling the underlying probability of error, unlike "Router Fit" metrics used in **Gen 67** (0.76 calibration).
- **Status Repair Saturation:** Status Repair Accuracy is consistently **1.000** across all analyzed programs (Gen 60-74). The logic `if repaired_flow > 0.5: status='up'` combined with "proven active" checks has effectively solved this sub-problem, shifting the optimization focus entirely to counter precision and confidence estimation.
- **Impact of Traffic Weighting:** **Gen 73** (0.91) showed that weighting anchors by traffic volume (`log`-scale) is effective, but **Gen 68** (0.92) showed that weighting by *peer agreement* ("Anchor Score") is slightly superior. Trusting a router because its neighbors agree is more robust than trusting it just because it processes a lot of traffic.

# META RECOMMENDATIONS

The following are actionable recommendations for the next program generations:

Based on the analysis of the current best program (Generation 68) and the global insights, here are 5 actionable recommendations for future program mutations:

1.  **Implement Device-Level Reliability Scores**
    Extend the "Anchor Score" concept (which currently evaluates trust per-link based on neighbors) to a global "Device Reliability" metric. In Phase 1, calculate the average agreement of *all* interfaces on a router with their peers. In Phase 2, use this global score to weight the `s_val` (Self) hypothesis against the `p_val` (Peer) hypothesis. This allows the solver to systematically discount telemetry from devices that are statistically noisy across the board, rather than treating every link as an isolated dispute.

2.  **Variance-Adaptive Flow Sigma**
    The current program dynamically scales the *weight* (height) of the flow hypothesis based on router balance, but uses a standard traffic-dependent `sigma` (width). You should also scale the `sigma` inversely to the router's balance score. If a router is currently imbalanced, the flow conservation target is inherently "fuzzy"; using a wider Gaussian (`sigma * (1.0 + (1.0 - balance))`) allows the solver to accept values that roughly balance the router without enforcing false precision.

3.  **Magnitude-Dependent Phantom Flow Penalty**
    Refine the Status-Aware Prior logic. Currently, the program applies a fixed `0.15` penalty to the `0.0` hypothesis if the interface is Up. You should scale this penalty based on the magnitude of the competing non-zero hypotheses (Self/Peer). If the alternative hypothesis suggests a massive flow (e.g., > 10Gbps), the prior probability of the `0.0` hypothesis should be practically zero (e.g., `0.01`). This prevents the model from "repairing" large, active flows to zero due to minor noise or conservation conflicts.

4.  **External Link "Trust Transfer"**
    For "Edge" links (interfaces without peers), the solver relies entirely on flow conservation. Enhance this by boosting the `f_weight` specifically for these links if the local router has a high "Anchor Score." If a router's internal links are highly consistent with their peers, the router is trustworthy; you should significantly increase the influence of the conservation constraint (`f_weight *= 2.0`) on its external links to force them to match the verified internal traffic.

5.  **Iterative Candidate Pruning (Annealing)**
    Implement a mechanism to reduce the search space in later iterations. If a candidate hypothesis (e.g., `s_val`) accumulates negligible probability mass (e.g., < 0.05) after the 3rd iteration, remove it from the `candidates` list for the remaining iterations. This "annealing" process prevents the solver from oscillating back to highly improbable values in the final steps and consolidates confidence mass onto the viable solutions.
